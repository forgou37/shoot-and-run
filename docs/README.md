# Документація гри «Shoot & Run»

Повний набір технічних і дизайнерських документів проєкту — TowerFall Ascension–**натхненна** (не клон) локальна та онлайн арена-стрілялка для 2–4 гравців на одному екрані.

Усі назви, арт та контент — оригінальні. Це жива документація: вона описує стан коду на гілці на момент написання. Канонічним джерелом правди для рішень залишається [`CLAUDE.md`](../CLAUDE.md) (Decisions Log) і теки [`specs/`](../specs/).

> Прозовий текст — українською; усі ідентифікатори, імена файлів, типи, ключі конфігів і команди — мовою коду (English), щоб документація лишалася точним дзеркалом репозиторію.

## Швидкий старт для нового розробника

```bash
npm install                 # монорепо на npm workspaces
npm run dev                 # Vite dev-сервер гри (http://localhost:5173)
npm test                    # усі Vitest-сюїти (sim + bots + net + game), headless у Node
npm run e2e                 # Playwright (shell + lobby + bots + online)
npm run dev:host            # локальний онлайн-хост (ws://localhost:8787)
```

URL-параметри для швидкого входу в гру (див. [Рендеринг](06-rendering-and-graphics.md)):

- `?quickstart=1` — матч 2 клавіатури, FFA
- `?bots=N&difficulty=hard` — людина проти N ботів
- `?online=ws://localhost:8787` — онлайн-клієнт (відкрий дві вкладки)
- `?rects=1` — дебаг-рендер кольоровими прямокутниками (по хітбоксах)

## Три речі, які треба знати про цей проєкт

1. **Симуляція — чиста й детермінована.** `packages/sim` не імпортує Phaser/DOM. Однакові `arena + tuning + seed + інпути` → побайтово однаковий лог подій. Це доводиться двома golden-фікстурами на кожному CI-прогоні (зокрема на Linux як cross-OS перевірка float-детермінізму).
2. **Тюнінг — це дані.** Кожне число геймфілу живе в одному файлі `content/tuning.json`. Захардкоджене тюнінг-число в коді — це баг за визначенням.
3. **Shell — тонкий.** Phaser лише перекладає ввід у структури `PlayerInput`, кличе `sim.step()` на фіксованому кроці 60 Гц і рендерить стан з інтерполяцією. Жодної ігрової логіки в рендер-колбеках.
